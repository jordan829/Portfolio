Space Sim

Project time frame: 2 weeks

This application is a simplistic space flight simulator created using C++ and OpenGL. It was created in a group of 3 for our final project in an intro to computer graphics course at UCSD as a way to showcase the engine that we had built up over the quarter. This engine includes various math libraries, lighting, a scene graph, and a parser/renderer for .obj files. For this project, I was personally responsible for implementing an l-system for procedurally generated plants and a particle effect system. The l-system is used to generate a tree at the center of the scene. The user can continuously generate a new plant by pressing the "T" key. The particle effect system was used mainly for creating the thruster effect on the spaceship. As the user increases the speed of the ship, the particles are set to travel at a higher velocity, creating a larger cone for a more realistic effect. An additional "Turbo Mode" feature was added, in which particles fly from the center of the screen outward around the ship, giving the illusion that you are traveling at light speed. Additionally, the particle system was used to create a procedurally generated star system. Each star is a single particle with a randomly generated position and lifetime, which gives the stars a "twinkling" effect.

Air Race

Project time frame: 2 weeks

This video game was developed in Unity3D using C# and is played using the Sony PlayStation Move and the PlayStation Eye. The core gameplay involves the user flying a UFO through a racetrack. This racetrack is set over a model of the UCSD campus and features checkpoints spread around the scene. The user controls the UFO exclusively with the Move controller. The speed of the UFO is dependent on how close the Move controller is to the Eye. The UFO can be turned by rotating the controller left and right (about the Y-Axis), and elevation can be controlled by raising and lowering the controller. When the game starts, there is a 3 second countdown before actual flight is enabled. Once the countdown ends, the timer starts. There are two different wayfinding aids to help the user find the next checkpoint. One is the red arrow that is displayed above the UFO. This will point to the next checkpoint and updates after each successive checkpoint is reached. The second aid is the distance text displayed on screen. This tells the user how far away from the next checkpoint they are. Once the user reaches the final checkpoint, the on screen timer stops.

Music App Prototype

Project time frame: 2 weeks

Project Overview

This is a prototype for an application providing music streaming and playlist creation. It was created to be an improvement upon existing apps such as Spotify or Apple Music. While working in a group of 3, I was personally responsible for conducting interviews, determining important user scenarios, task analysis, and interface design.

Interviews

The process of creating this prototype began with our group conducting interviews of other students who regularly use music streaming apps. I personally conducted 4 interviews between friends and other UCSD students. This project was mainly focused on the user's experience while creating and managing their personal playlists, and our questions reflected this. We aimed to determine what tasks and scenarios are most regularly performed by users.

User Scenarios

After conducting the interviews, we then had our interviewees walk us through 3 different scenarios that reflected their responses during the interview. For example, one interviewee mentioned that he sometimes likes to search for artists similar to artists that he already knows of and enjoys. After the interview, I had him walk me through the process of finding such information, taking screenshots of each action he performs. After all of the interviews were conducted, we tallied up all of our user scenarios and then performed full walkthroughs of the five most popular.

Task Analysis

Once we determined which scenarios were the most common, we began our task analyses. These analyses consisted of 3 walkthroughs per scenario, each within a separate music app. These apps included Spotify, Apple Music, Pandora, and Soundcloud. We charted the number of actions performed to determine which apps cater to these tasks the best. We used the results from these analyses to help guide us during our own application design process.

Prototyping

After completing our research, we began the stages of prototyping our app. I was made personally responsible for designing the majority of the user interface. I used Balsamiq to create images for individual pages and menus. These included the home page, the playlists page, drop down menus, and individual artist/album pages. While creating these pages, I had to keep in mind how to improve our app and make it more user friendly. During this process, I had to make many hard decisions, the most notable of which are listed below.

What to use for the home page?

I decided to give users direct access to playlists, library, and browse upon launching the app. These seemed to be the most popular sections across all music apps. Additionally, we concluded amongst our group that the information included on home pages for the other apps that we analyzed were not very useful for the average user.

How to streamline the process of adding to playlists?

I noticed from our conducted interviews that many users only had 1 playlist or a main playlist that they would continuously update with new songs. To appeal to these users, I decided to add a "Main Playlist" feature. User's can select which playlist they want to be their main playlist. When adding a song they are given the option to "Add to Main Playlist" instead of having to find this playlist in a separate list.

How to organize user's music library?

I realized that users prefer to have their music library displayed to them in different ways. Some users prefer to have their music sorted by artist, while others prefer to have just a list of all of their saved songs. To make this easier, I included a drop down menu on the "My Music" page that displays these different options. Ideally, the app would save whichever setting the user chose last so that they would not have to make the same decision each time they view their library.

3D Graph Visualizer

Project time frame: 2 weeks

This application was developed in Unity3D using C# and is controlled using the Leap Motion controller. The application reads in graph information (nodes and edges) from a .gml file and draws the graph using a force directed algorithm. The user can manipulate the graph using a range of hand gestures. For example, pinching and moving of a single hand will rotate the graph as if controlled by a trackball. Pointing with a single finger and moving your hand will control the cursor. There are also 3 different methods of selection. Single selection, in which a single node can be selected. Multi selecion, in which a volume of nodes can be selected at once. And sequential selection, in which multiple nodes can be selected one after another. Once the user selects whichever nodes they desire, they may perform the following. Swiping with one hand will delete the node/nodes from the graph. Tapping with one finger will highlight the edges to all connected nodes. Additionally, holding a closed fist for a number of seconds will bring a single selected node centered in front of the camera.

Map Maker

Project time frame: 2 weeks

This application is a level editor created using Unity3D and C# and is controlled by using the zSpace. The user can select from a variety of objects displayed at the side of the screen to be placed into the map grid. Once an object is placed into the grid, it is snapped into place according to the nearest cell. This ensures that all objects are evenly placed and eases the process of aligning objects. After the map is created, the user can click on play (right side of screen) to begin the demo mode. In this mode, the player object can be controlled by clicking on a point in the map, causing the player to move in that direction. Additionally, double clicking causes the player to perform a jump, allowing more vertical maps to be created. Once the player reaches the donut, which acts as the goal, text is displayed in the center of the screen indicating a win condition. If the player falls off into the lava below, different text is displayed indicating a loss.

Space Shooter

Project time frame: < 1 week

This application is an orthographic 3D video game built using C++ and OpenGL. The objective of the game is simple: Shoot stuff for points. If an enemy ship passes, the player loses one life. Shooting red enemies rewards the player with one extra life, with a maximum life count of 5. Red enemies begin to spawn slightly more frequently as the player loses more lives. Collision detection is performed by calculating the magnitude of the vector between a missile and an enemy. If the missile is determined to be inside of the ship's collider, both the missile and the enemy ship are destroyed and a particle emitter is spawned at the enemy ship's location, indicating a hit. The particle emitter interpolates the particle colors from white to orange over time.

VReakout

Lead Developer

3D User Interface

Project time frame: 10 weeks

Project Overview

The goal of this project was for our team to create a generic 3D user interface prototype for use within virtual reality applications using dual hand controllers. The prototype was developed for the HTC Vive using Unity 3D but we purposefully designed the interface in a way so that when other dual-controller head-mounted displays become available, the interface can be easily ported. Although we wanted the interface to be for generic applications, we based the interface around a visualization tool that is able to render 3D textures of CT scan data. The interface at this stage displays a menu hierarchy. Menus are created using xml and allow users to organize widgets and actions into folders to later be accessed. The main interface is split into two parts: a spherical helmet menu, and a tool belt menu system. The helmet menu follows the user’s head position and serves as the main menu and has access to the entire menu hierarchy. Particle System

Project time frame: 2 weeks

A program that emits a particle system that is affected by uniform gravity and simple ground plane collisions. Particles are drawn as 2D billboard polygons and can be varied in size and color. Additionally, there are simple controls to move the system around, adjust emission properties, and adjust the ambient wind speed.

Cloth Simulation

Project time frame: 2 weeks

A program that simulates a piece of cloth made from particles, spring-dampers, and triangular surfaces. It includes the effects of uniform gravity, spring elasticity, damping, aerodynamic drag, and simple ground plane collisions. Some particles of the cloth are ‘fixed’ in place in order to hold the cloth up, and there are simple controls to move these fixed points around, as well as controls to adjust the ambient wind speed.
